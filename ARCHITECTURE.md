# Investment Analysis — 설계 문서

## 프로젝트 개요

한국 주식 종목에 대한 투자 분석을 자동화하는 시스템. 기술적 지표, 재무제표, 뉴스 감성, 머신러닝 예측을 결합해 최종 투자 의견을 생성한다.

---

## 데이터 흐름

```
pykrx          → 주가 OHLCV          → 기술지표(RSI·MACD·볼린저) ─┐
                                                                    ├→ LSTM 예측 ─┐
OpenDART       → 재무제표            → 텍스트 임베딩(1536-d)      ─┘             │
                                    → TTM 밸류에이션(PER·PBR·ROE) ──────────────┤
                                                                                  ├→ LLM 종합
NewsAPI        → 뉴스 기사          → GPT 감성 분석(호재·악재)   ──────────────┘
                                                                       ↓
                                                            매수·관망·매도 추천
```

---

## 설계

### 1. 멀티모달 LSTM

기술적 지표(숫자 시계열)만으로 주가를 예측하는 모델은 "숫자 패턴"밖에 보지 못한다. 같은 차트 패턴이라도 삼성전자와 중소형주의 의미는 다르고, 직전 분기 실적이 어땠는지에 따라 상승의 지속성이 달라진다.

두 가지 신호를 하나의 모델 안에서 결합한다:

- **LSTM Branch**: 20거래일 기술지표 시퀀스 → 128-d hidden state
- **Text Branch**: DART 재무요약 텍스트 → OpenAI 임베딩(1536-d) → 64-d projection

두 표현을 concat해 상승/하락/횡보 3분류를 예측한다. 텍스트 임베딩이 "이 회사가 어떤 회사인지"라는 맥락을 제공해 기술지표 해석의 배경이 된다.

```
LSTM(seq_20 × 11_features) → 128-d
                                       → Fusion(192-d) → 3-class
Text(1536-d) → Dense → 64-d
```

**개선 가능성**
- Text Branch의 임베딩이 학습 시점에 고정된다. 분기마다 재무 수치가 바뀌어도 모델을 재학습하기 전까지 임베딩은 그대로다. 추론 시점에 최신 재무 텍스트를 매번 임베딩하도록 변경하면 이 문제를 해소할 수 있다.
- 현재 두 branch를 단순 concat으로 합친다. Cross-attention으로 교체하면 "어떤 기술지표가 이 재무 맥락에서 더 중요한지"를 모델이 스스로 학습할 수 있다.

---

### 2. 3분류 레이블 (상승/하락/횡보)

이진 분류(상승/하락)는 횡보 구간을 한쪽으로 억지로 분류한다. 실제 시장에서 종목이 ±2% 이내를 오가는 구간이 전체 거래일의 상당 부분을 차지하며, 이 구간을 "상승"이나 "하락"으로 레이블링하면 학습 노이즈가 심해진다.

`threshold=0.02`(2%)를 기준으로 횡보를 별도 클래스로 분리해 모델이 "불확실 구간"을 명시적으로 학습한다.

**개선 가능성**
- threshold 2%는 고정값이다. 변동성이 낮은 대형주(삼성전자)와 높은 중소형주에 같은 기준을 적용하는 것은 불합리하다. 종목별 ATR(Average True Range) 기반으로 동적 threshold를 계산하는 방향을 고려할 수 있다.
- 현재 레이블은 `pred_days=5`(5거래일) 후 단일 시점의 수익률만 본다. 5일 중 최대 낙폭(MDD)을 함께 레이블에 반영하면 리스크 관점도 학습할 수 있다.

---

### 3. 시퀀스 길이(seq_len=20)와 예측 horizon(pred_days=5)

- `seq_len=20`: 약 1개월치 거래일. 기술적 분석에서 20일 이동평균(SMA-20)이 단기 추세의 기준선으로 쓰이기 때문에, 모델이 보는 시퀀스 길이를 같은 스케일로 맞췄다.
- `pred_days=5`: 약 1주일 후 예측. 당일 예측은 노이즈 비율이 높고, 한 달 이상은 단기 기술지표의 유효성이 낮아진다. 1주일이 기술적 신호와 예측 horizon의 균형점으로 판단했다.

**개선 가능성**
- 두 파라미터가 현재 `config.py`에 고정되어 있다. 단기(5일)/중기(20일) 예측을 동시에 학습하는 멀티-horizon 출력 헤드를 추가하면, 사용자가 투자 기간에 따라 예측 결과를 선택할 수 있다.

---

### 4. 재무 수치를 텍스트 임베딩으로 인코딩

DART API에서 받은 재무 수치(매출, 순이익, ROE 등)를 숫자 그대로 모델에 넣을 수도 있었다. 텍스트로 직렬화해 OpenAI 임베딩을 쓴 이유:

- 재무 수치는 업종마다 스케일이 다르다(반도체 매출 300조 vs 소형 유통사 500억). 직접 정규화하면 업종 간 비교 정보가 손실된다.
- 사전학습된 언어 모델의 임베딩 공간은 "삼성전자 반도체"와 "TSMC"가 가깝게 위치한다. 수치 피처만으로는 이런 의미 유사성을 인코딩하기 어렵다.
- 텍스트 포맷이라 임베딩 입력에 새로운 재무 항목을 추가해도 모델 구조를 바꿀 필요가 없다.

**개선 가능성**
- 현재 임베딩 텍스트가 단순 수치 나열이다(`매출 300조, ROE 15%`). 재무 변화 추이(전년 대비 성장률)나 업종 내 순위 같은 상대적 정보를 포함하면 임베딩 품질이 높아진다.
- API 비용이 발생한다. 재무 수치 변화가 없는 경우 기존 임베딩을 캐싱해 재사용하는 전략이 필요하다.

---

### 5. 단계별(Stage-based) 파이프라인

전체 분석을 하나의 함수로 실행하면 어느 단계에서 실패했는지 알기 어렵고, 중간 결과를 볼 수 없으며, 특정 단계만 재실행하기도 어렵다.

8단계로 분리해 각 단계가 독립적으로 실행·확인·재시도 가능하도록 설계했다. Streamlit UI의 탭 구조가 이 설계에서 자연스럽게 도출된다. 각 탭은 해당 단계의 실행 상태(대기/진행/완료/오류)를 시각적으로 표시한다.

**개선 가능성**
- 현재 단계들이 순차 실행된다. 재무 데이터 수집(DART)과 뉴스 수집(NewsAPI)은 서로 의존성이 없으므로 병렬 실행이 가능하다. `asyncio` 또는 `ThreadPoolExecutor`로 전환하면 전체 분석 시간을 단축할 수 있다.
- 각 단계의 결과가 `st.session_state`에만 저장된다. 브라우저를 새로고침하면 모든 결과가 사라진다. 단계별 결과를 로컬 파일이나 DB에 저장하면 세션 복원이 가능해진다.

---

### 6. LLM 최종 종합

수치 기반 규칙(예: "RSI < 30이고 MACD 골든크로스면 매수")은 조건 조합이 복잡해질수록 유지보수가 어렵고, 예외 상황에 대응하기 어렵다. 모든 분석 신호를 GPT 프롬프트로 넘기고 자연어로 종합하게 하면:

- 신호 간 상충 관계를 맥락적으로 해석한다(예: 기술적으로는 매수 신호지만 뉴스 리스크가 크다)
- 규칙 테이블 없이도 다양한 시장 상황에 대응한다
- 사람이 읽을 수 있는 형태로 출력된다

단, LLM은 hallucination 가능성이 있으므로 최종 판단 도구가 아닌 **참고 자료**로만 사용한다.

**개선 가능성**
- 현재 프롬프트가 모든 신호를 동등하게 나열한다. 신뢰도가 높은 신호(예: LSTM confidence > 80%)에 더 높은 가중치를 부여하는 structured prompt를 설계하면 분석 품질이 높아진다.
- 사이드바에서 GPT 모델을 선택할 수 있는 UI가 있지만 실제 API 호출에 반영되지 않는 버그가 있다. `st.session_state.gpt_model`을 `analyze_with_llm()`에 파라미터로 전달해야 한다.

---

### 7. OpenDART 직접 연동

재무 데이터를 제공하는 유료 API가 더 편리하지만, OpenDART는 금융감독원이 운영하는 공식 공시 시스템으로 국내 상장 기업의 사업보고서·반기보고서를 무료로 제공하며 데이터 신뢰성이 높다. API 호출 제한이 있어 `time.sleep(0.3)`으로 속도를 제어한다.

**개선 가능성**
- 현재 기업코드 목록(`corpCode.xml`)을 분석마다 다운로드한다. 하루 단위로 캐싱하면 불필요한 네트워크 요청을 줄일 수 있다.
- 반기·연간 보고서만 수집한다. 분기 보고서(11013, 11014)까지 포함하면 더 최신 재무 상태를 반영할 수 있다.

---

### 8. 주가 데이터 증분 CSV 캐싱

pykrx를 통한 KRX 데이터 수집은 느리고 반복 호출 시 서버 부하가 발생한다. 한 번 수집한 데이터는 `data/downloads/{code}.csv`로 저장하고, 이후 요청 시 누락된 기간만 추가 수집하는 증분 전략을 사용한다.

**개선 가능성**
- 현재 캐시 유효성 검사가 날짜 범위 기반이다. 수정 주가(액면분할, 배당락) 반영 여부를 확인하는 로직이 없어, 분할이 발생한 종목의 과거 데이터가 오염될 수 있다.
- CSV는 단일 파일이라 여러 프로세스가 동시에 같은 종목을 분석할 때 충돌 위험이 있다. SQLite 같은 경량 DB로 교체하면 동시성 문제를 해결할 수 있다.

---

### 9. 뉴스 감성 분석 (신규 추천)

현재 뉴스 분석은 NewsAPI로 수집한 헤드라인을 GPT에 넘겨 호재/악재를 분류한다. 헤드라인만 보는 방식은 기사 본문의 맥락을 놓친다.

공시 텍스트(DART 사업보고서 주요 내용)를 함께 분석 대상에 포함하면, 실적 발표나 대규모 계약 같은 이벤트를 뉴스보다 빠르게 포착할 수 있다. DART는 공시 원문을 제공하므로 추가 크롤링 없이 연동 가능하다.

---

### 10. 백테스트 (신규 추천)

현재 LSTM 모델의 성능 지표는 검증 셋 정확도(accuracy)뿐이다. 분류 정확도가 높더라도 실제 수익률과 무관할 수 있다. "상승 예측 시 매수, 하락 예측 시 관망"하는 단순 전략을 과거 데이터에 적용해 수익률·MDD·샤프 지수를 측정하는 백테스트 모듈이 필요하다.

과거 KRX 데이터는 이미 수집·저장되어 있으므로, 수집 파이프라인을 재사용해 백테스트 데이터를 구성할 수 있다.

## 미결 이슈

| 항목 | 내용 |
|------|------|
| GPT 모델 선택 버그 | 사이드바 UI에서 선택한 모델이 실제 API 호출에 반영되지 않음 |
| LSTM 재학습 주기 | 모델이 특정 시점 데이터로 학습된 후 고정됨. 주기적 재학습 전략 없음 |
| 실시간 데이터 | 일봉 데이터만 사용. 장중 분석 불가 |
| 테스트 코드 없음 | 단위 테스트·통합 테스트가 없어 리팩토링 시 회귀 탐지 어려움 |
