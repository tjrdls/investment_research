# Investment Analysis — 설계 문서

> 각 섹션은 PPT 슬라이드 1장에 대응한다.
> 포맷: **기존 방식 → 문제점 → 개선/결정 → 이유** 순으로 통일.

---

## 0. 프로젝트 개요

한국 주식 종목에 대한 투자 분석을 자동화하는 시스템.
기술적 지표, 재무제표, 뉴스 감성, 머신러닝 예측을 결합해 **LLM이 최종 투자 의견을 생성**한다.

```
pykrx      → 주가 OHLCV  → 기술지표(RSI·MACD·볼린저) ───┐
                                                          ├→ LSTM 예측 ──┐
OpenDART   → 재무제표    → 텍스트 임베딩(1536-d) ────────┘              │
           → 공시 목록   → 임베딩 컨텍스트 추가                          ├→ LLM 종합
                         → TTM 밸류에이션(PER·PBR·ROE) ────────────────┤
NewsAPI    → 뉴스 기사   → GPT 감성 분석(호재·악재) ───────────────────┘
                                                                ↓
                                                     매수 / 관망 / 매도
```

---

## 1. 모달리티 결합 방식 — Intermediate vs Late Fusion

### 선택지

**Intermediate Fusion**

```
LSTM(tech_seq) → 128-d ─┐
                          ├─ concat → MLP → 3-class
Text_embed     →  64-d ──┘
```

모델 내부에서 기술 + 재무 정보를 결합해 하나의 분류기로 출력.

**Late Fusion**

```
LSTM(tech)  → 상승 확률 0.68
기술지표    → 매수 신호
뉴스        → 악재
재무        → 텍스트
           ↓
         LLM이 종합 판단
```

각 모달리티가 독립적으로 신호를 생성하고, LLM이 이를 해석해 판단.

---

### 핵심 고민

> **"정보를 모델 안에서 섞을 것인가, 밖에서 해석하게 할 것인가"**

---

### 결정에 가장 큰 영향을 준 요소

Intermediate Fusion에서 LLM이 받는 정보:

```
→ "상승 확률 0.68"
```

왜 상승인지, 어떤 신호가 영향을 줬는지 알 수 없다. 이미 압축된 결과물이다.

Late Fusion에서 LLM이 받는 정보:

```
→ LSTM: 상승 확률 0.68 (confidence 72%)
→ 기술지표: RSI 정상, MACD 골든크로스 → 매수 신호
→ 뉴스: 반도체 업황 악화 → 악재
→ 밸류에이션: PER 12x (역사적 하위 25%)
```

신호 간 충돌을 인식하고 reasoning을 생성할 수 있다.

---

### 설계 철학 비교

| 구분           | Intermediate Fusion | Late Fusion            |
| -------------- | ------------------- | ---------------------- |
| 역할           | 모델이 판단         | 모델 + LLM이 분산 판단 |
| 정보 구조      | 압축됨 (블랙박스)   | 분리됨 (해석 가능)     |
| 해석 가능성    | 낮음                | 높음                   |
| LLM 활용       | 제한적              | 극대화                 |
| 신호 충돌 처리 | 불가                | 가능                   |

---

### 최종 결정: Late Fusion 철학 채택

**Intermediate Fusion을 채택하지 않은 이유**

- 내부 결합으로 의미가 압축되어 LLM이 활용할 수 없다
- 어떤 신호가 영향을 줬는지 해석 불가능하다
- "모델이 이미 판단을 완료"하는 구조는 LLM의 역할을 축소시킨다

**Late Fusion을 채택한 이유**

- 각 신호가 분리되어 LLM이 reasoning 가능하다
- "기술적 매수 + 뉴스 악재" 같은 충돌을 LLM이 해석할 수 있다
- 새로운 신호(공시, 거시지표 등)를 추가하기 쉽다
- Late Fusion의 단점(신호 간 상호작용을 모델이 학습 못함)은 LLM이 대신 해석하도록 설계해 보완한다

---

### 최종 구현 구조 (Hybrid Late Fusion + LLM 해석)

```
[신호 생성]
  LSTM(기술지표 시퀀스)  → 상승/하락/횡보 확률 + confidence
  기술적 분석 규칙        → 신호 목록 + 스코어
  뉴스 GPT 분석          → 호재/악재 판정 + 점수
  재무제표 + 공시 텍스트  → 임베딩 컨텍스트 + TTM 밸류에이션

[판단]
  LLM                    → 모든 신호를 해석해 매수/관망/매도 결론
```

> **핵심:** 이 프로젝트는 모델을 똑똑하게 만드는 게 아니라, LLM이 똑똑하게 생각할 수 있게 만드는 설계다.

---

### 구현 완료

Text Branch(Intermediate Fusion 요소)를 제거하고 `StockPredictor`로 재명명했다. 기술지표 시퀀스만 입력으로 사용하며, 재무 텍스트 임베딩은 LLM 컨텍스트로만 전달된다. 코드와 설계 철학이 일치한다.

**남은 과제**
Cross-attention 구조를 도입하면 LSTM 내부에서 시점별로 "어떤 지표가 지금 더 중요한지"를 학습할 수 있다. 이는 Intermediate Fusion이 아닌 self-attention이므로 Late Fusion 철학과 충돌하지 않는다.

## 2. LSTM 학습 품질 개선

**기존 방식**

- 데이터를 학습 80% / 검증 20%로 분리
- 손실 함수: `CrossEntropyLoss` (가중치 없음)
- 평가 지표: Accuracy만 출력

**문제점**

- Test set이 없어 학습에 사용되지 않은 데이터에서의 실제 성능을 알 수 없다. LSTM 출력이 LLM 입력으로 사용되므로 신뢰할 수 없는 모델이 파이프라인 전체에 영향을 준다.
- 상승/하락/횡보 분포가 불균형하면 다수 클래스만 잘 맞추고 소수 클래스를 무시하는 방향으로 수렴한다.
- Accuracy가 높아도 "하락"을 전혀 잡지 못하는 모델일 수 있다.

**개선 방식**

- **Train / Val / Test = 70 / 15 / 15** 분리. Test set은 학습·검증 과정에서 완전히 격리.
- **WeightedCrossEntropyLoss**: 학습 데이터의 클래스 분포를 자동 계산해 `(1/cls_cnt)*n/3` 가중치 적용.
- **classification_report**: 학습 완료 후 Test set에서 상승·하락·횡보 각각의 Precision / Recall / F1 출력.

**이유**
Test set은 모델이 한 번도 본 적 없는 데이터다. 이 셋에서의 F1이 실제 서비스 품질에 가장 가깝다. 가중 손실은 소수 클래스(예: 하락)의 신호 강도를 유지해 LLM이 "하락 경고"를 받지 못하는 문제를 방지한다.

## 3. 3분류 레이블 (상승 / 하락 / 횡보)

**기존 방식 (대안)**
이진 분류: 상승(1) / 하락(0).

**문제점**
실제 시장에서 종목이 ±2% 이내를 오가는 횡보 구간이 전체 거래일의 상당 부분을 차지한다. 이 구간을 억지로 상승이나 하락으로 분류하면 학습 노이즈가 심해진다.

**설계 결정**`threshold=0.02`(2%)를 기준으로 3분류:

- 수익률 > +2% → 상승 (class 0)
- 수익률 < −2% → 하락 (class 1)
- 그 외 → 횡보 (class 2)

**이유**
횡보를 별도 클래스로 분리하면 모델이 "불확실 구간"을 명시적으로 학습한다. LLM이 "LSTM이 횡보를 예측했다"는 정보를 받으면 보수적 판단(관망)을 유도할 수 있다.

**남은 과제**
threshold 2%는 전 종목 고정값이다. 변동성이 낮은 대형주와 높은 중소형주에 같은 기준을 적용하는 것은 불합리하다. 종목별 ATR(Average True Range) 기반으로 동적 threshold를 계산하는 방향을 고려할 수 있다.

## 4. 시퀀스 파라미터 설계 (seq_len=20, pred_days=5)

**설계 결정**

- `seq_len=20`: 약 1개월치 거래일을 LSTM 입력 시퀀스로 사용
- `pred_days=5`: 5거래일(약 1주일) 후 수익률을 예측 대상으로 설정

**이유**

- 20일 이동평균(SMA-20)이 기술적 분석에서 단기 추세의 기준선으로 쓰인다. 모델이 보는 시퀀스 길이를 같은 스케일로 맞춰 기술지표와 일관성을 확보했다.
- 당일(1일) 예측은 노이즈 비율이 지나치게 높다. 한 달(20일) 이상은 단기 기술지표의 유효성이 낮아진다. 1주일이 기술적 신호의 유효 기간과 예측 horizon의 균형점이다.

**남은 과제**
두 파라미터가 `config.py`에 고정되어 있다. 단기(5일) / 중기(20일) 예측을 동시에 학습하는 멀티-horizon 출력 헤드를 추가하면, 투자 기간에 따라 예측 결과를 선택할 수 있다.

## 5. 재무 수치를 텍스트 임베딩으로 인코딩

**기존 방식 (대안)**
DART에서 받은 재무 수치(매출, 순이익, ROE 등)를 숫자 피처로 그대로 모델에 입력.

**문제점**

- 재무 수치는 업종마다 스케일이 다르다. 반도체 기업 매출 300조와 소형 유통사 매출 500억을 같은 공간에서 정규화하면 업종 간 비교 정보가 손실된다.
- 새로운 재무 항목(예: 연구개발비)을 추가하려면 모델 입력 차원을 바꿔야 한다.

**설계 결정**
재무 수치를 자연어 텍스트로 직렬화 후 OpenAI 임베딩(1536-d) 적용.

```
"삼성전자 | [2024년 연간] 매출 3000억 순이익 150억 ROE 12.5%
           | 최근공시: 사업보고서 | 반기보고서 | ..."
```

**이유**
사전학습된 언어 모델의 임베딩 공간에서 업종이 비슷한 기업들은 자연스럽게 가깝게 위치한다. 텍스트 포맷은 새로운 재무 항목이나 공시 정보를 추가해도 모델 구조를 바꿀 필요가 없어 확장성이 높다.

**남은 과제**
현재 텍스트가 수치 나열에 가깝다. 전년 대비 성장률이나 업종 내 순위 같은 상대적 정보를 포함하면 임베딩 품질이 높아진다. 재무 수치가 변하지 않은 경우 임베딩을 캐싱해 API 비용을 줄일 수 있다.

## 6. OpenDART 연동 품질 개선

**기존 방식**
연도 범위를 고정 루프로 순회하며 보고서를 요청. 당기순이익이 0으로 조회되면 그대로 사용.

**문제점**

- 아직 제출되지 않은 연도의 보고서를 요청하면 DART API가 빈 결과를 반환한다. 불필요한 API 호출과 데이터 누락이 발생한다.
- 연결재무제표 구조에 따라 "당기순이익" 항목이 0으로 잡히는 기업이 있다. 실제 이익이 "지배주주지분 당기순이익" 항목에 기록되어 있어도 0으로 수집된다.
- 임베딩 컨텍스트에 재무 수치만 포함되어 최근 공시(실적 발표, 대규모 계약 등) 정보가 빠진다.

**개선 방식**

- **`find_available_reports()`**: DART 공시 목록 API로 실제 제출된 보고서만 조회한 뒤 해당 보고서만 요청. 고정 루프 대신 "있는 것만 가져온다".
- **지배주주지분 fallback**: 당기순이익이 0이면 지배기업 소유주지분 항목으로 재시도.
- **`get_recent_disclosures()`**: 최근 공시 제목 5건을 조회해 임베딩 텍스트에 추가.

**이유**
데이터 품질이 LSTM 입력과 LLM 컨텍스트 모두에 직접 영향을 준다. 잘못된 재무 수치(0)가 임베딩에 포함되면 모델이 잘못된 맥락으로 예측하고, LLM은 그 예측을 신뢰 있는 신호로 받아들인다. 공시 텍스트는 뉴스보다 빠르게 이벤트를 포착할 수 있다.

## 7. 단계별(Stage-based) 파이프라인

**기존 방식 (대안)**
전체 분석을 하나의 함수에서 순차 실행.

**문제점**
어느 단계에서 실패했는지 알기 어렵고, 중간 결과를 확인할 수 없으며, 특정 단계만 재실행하기도 어렵다.

**설계 결정**
8단계로 분리. 각 단계는 독립적으로 실행·확인·재시도 가능하다.

```
1. 주가 수집  →  2. 기술지표  →  3. 재무·공시  →  4. 텍스트 임베딩
→  5. LSTM 예측  →  6. 뉴스 감성  →  7. 밸류에이션  →  8. LLM 종합
```

Streamlit UI의 탭 구조가 이 설계에서 자연스럽게 도출된다. 각 탭은 해당 단계의 실행 상태(대기 / 진행 / 완료 / 오류)를 시각적으로 표시한다.

**이유**
파이프라인이 중간에 실패해도 성공한 단계의 결과는 `session_state`에 남아 있다. 재무 API 오류 시 이전 단계(주가·기술지표)까지의 결과로 부분 분석이 가능하다.

**남은 과제**
재무 수집(DART)과 뉴스 수집(NewsAPI)은 서로 의존성이 없어 병렬 실행이 가능하다. `asyncio` 또는 `ThreadPoolExecutor`로 전환하면 전체 분석 시간을 단축할 수 있다.

## 8. LLM 최종 종합 및 구조화 프롬프트

**기존 방식**
모든 신호를 동등하게 나열한 단일 프롬프트 → GPT가 자유롭게 판단 → 비구조화 텍스트 출력.

**문제점**

- 신뢰도가 낮은 LSTM 예측(confidence 30%)과 높은 예측(80%)을 같은 비중으로 다루면 LLM이 약한 신호에 휘둘릴 수 있다.
- 기술적 매수 신호와 뉴스 악재가 동시에 존재할 때 LLM이 어느 쪽을 따랐는지 출력에서 알 수 없다.

**개선 방식**

- **신호 가중 프롬프트**: LSTM confidence 수준에 따라 프롬프트 내 강조 문구를 자동 조정한다.
  - ≥ 70%: "고신뢰 신호 — LSTM 예측을 핵심 근거로 활용하세요"
  - 50~70%: "중간 신뢰 — 다른 신호와 교차 확인하세요"
  - < 50%: "저신뢰 — 기술·뉴스 신호를 우선하세요"
- **충돌 자동 감지**: LSTM 방향·기술 방향·뉴스 방향을 비교해 충돌 시 보수적 판단 유도 문구를 삽입한다.
- **구조화된 JSON 출력** 확장:

```json
{
  "recommendation": "매수 / 관망 / 매도",
  "signal_interpretation": "각 신호를 어떻게 해석해 결론에 반영했는지",
  "conflict_resolution": "신호 충돌이 있다면 어떻게 판단했는지",
  "risks": [...],
  "opportunities": [...],
  "confidence": 0~100
}
```

**이유**
LLM의 맥락적 해석 능력은 유지하되, 입력 구조를 통제해 판단의 일관성을 높인다. `signal_interpretation`과 `conflict_resolution` 필드는 LLM이 어떤 근거로 결론을 냈는지를 기록해 후속 검증을 가능하게 한다.

**남은 과제**
사이드바에서 선택한 GPT 모델이 실제 API 호출에 반영되지 않는 버그가 있다. `st.session_state.gpt_model`을 `analyze_with_llm()`에 파라미터로 전달해야 한다.

## 9. 주가 데이터 증분 CSV 캐싱

**기존 방식 (대안)**
분석할 때마다 pykrx로 전체 기간 재수집.

**문제점**
pykrx를 통한 KRX 데이터 수집은 느리고, 반복 호출 시 서버 부하가 발생한다.

**설계 결정**
한 번 수집한 데이터는 `data/downloads/{code}.csv`로 저장하고, 이후 요청 시 **누락된 기간만 추가 수집**하는 증분 전략을 사용한다.

**이유**
동일 종목을 여러 번 분석하거나 여러 종목을 일괄 학습할 때 수집 시간이 선형으로 증가하는 문제를 방지한다. 증분 수집으로 기존 데이터를 재사용하면서 최신 거래일만 추가한다.

**남은 과제**
수정 주가(액면분할, 배당락) 반영 여부를 확인하는 로직이 없어, 분할이 발생한 종목의 과거 데이터가 오염될 수 있다. CSV 단일 파일 구조는 다중 프로세스 동시 접근 시 충돌 위험이 있다.

## 10. 밸류에이션 피처 설계 결정

**기존 방식 (대안 A — Colab)**
pykrx에서 일별 PER/PBR/EPS/BPS를 수집해 LSTM 피처(총 27개)로 직접 추가.

**기존 방식 (대안 B — 현재)**
밸류에이션을 LSTM에서 배제하고, LLM 단계에서만 참고 지표로 제공.

**문제점 (A)**

- 피처 27개 → 소량 학습 데이터에서 과적합 위험
- pykrx 시장 PER ≠ DART 재무제표 기반 계산값 (데이터 출처 혼재)

**문제점 (B)**

- 모델이 "저PER + 기술적 매수 신호" 같은 펀더멘털-기술 조합을 직접 학습하지 못한다

**설계 결정 (채택)**원시 수치 대신 **해석된 밸류에이션 신호**만 LSTM 피처에 추가한다.

- **PER 백분위**: 동일 종목의 과거 PER 분포에서 현재 PER의 위치 (0~1)
- **저평가 flag**: PER 백분위 < 0.25이면 1

**이유**
"PER = 8.3"은 업종에 따라 저평가일 수도 고평가일 수도 있다. 백분위로 변환하면 업종 무관하게 "역사적으로 얼마나 싼가"라는 의미 있는 신호가 된다. 피처 수를 최소로 유지해 과적합을 방지하고, LLM이 해석 가능한 형태로 신호를 제공한다.
